# 小羽模型训练方案

> 目标不是训一个更强的模型——是先证明"信念对齐的端侧小伙伴"真的感觉不一样。

---

## 一、护城河在哪儿

Nous 在每一条他们擅长的轴上都能碾我——算力、数据量、工程规模。

但他们**结构上造不出来**的东西是：

> **一个小的、带信念的、能离线陪你、越用越懂你的伙伴。**

因为 Nous 整个 thesis 是"中立、可 steer、云端、agentic 的工具模型"。他们的 Agent 没有自我，每次对话是全新的上下文。

而小羽有 CogniKernel——跨 session 的信念、反思、成长。

**这就是唯一的护城河。不跟 Hermes 比工具调用精度，比的是"陪伴感和信念一致性"。**

---

## 二、V1 最小路径：先证真，再建厂

### 做什么（1-2 周，1×4090，几乎不烧钱）

| 步骤 | 内容 | 成本 |
|------|------|------|
| 1 | 选现成小 instruct 模型（Qwen2.5-1.5B，真要多模态再上 MiniCPM） | 0 |
| 2 | LoRA SFT，只对齐两件事：信念表达 + 工具格式 | 1 周，1×4090 |
| 3 | 冷启动数据老实承认是蒸馏：用强 teacher（Qwen/DeepSeek）按信念快照生成 | 0 |
| 4 | 不做 RL，不做 PPO，不做在线学习 | 省 5000+ 美金 |
| 5 | 唯一要测的：一个单盲对比测试（见下文） | 0 |

### 不做什么（砍掉）

- ❌ PPO / GRPO / 多轮 RL — 等有用户、有真实 reward 来源再说
- ❌ 在线 RL / 持续学习 — 同上
- ❌ 端云信念同步（CRDT 难题，以后再说）
- ❌ 多模态保持 benchmark — v1 先做语言对齐
- ❌ 1.3B 自己端到端多步工具调用 — 降级为"识别意图 + 转交 Planner 执行"

### 工具调用降级策略

1.3B 做可靠多步工具调用是个伪命题——Hermes 2 Pro 能成是因为它 7B+。1.3B 学得稳"格式"，学不稳"决策"。

```
小羽端侧模型:
  用户说"帮我查明天天气顺便提醒我带伞"
  → 识别意图: "查询天气 + 设置提醒"
  → 格式化工具调用
  → 转交 Planner 执行
  → 根据 Planner 返回结果生成回复

不要求: 模型自己维护调用链、自己解析返回、自己错误恢复
```

v1 的底线：**格式对、意图准、回复与信念一致。** 至于"多步拆解"，那是 Planner / 云端的事。

## 三、数据策略：老实承认是蒸馏

### 数据来源

格数据不来自"海量用户"——我是 solo，量不够。冷启动就是蒸馏：

```
强 teacher (Qwen / DeepSeek) + 信念快照
  → 生成 <用户输入, 信念状态, 响应> 三元组
  → 人工校验质量
  → LoRA SFT 蒸馏进小羽
```

承认：v1 的小羽是"把 teacher 的信念扮演压缩进端侧"，天花板就是 teacher。

**这不丢人。** v1 够用就好——目标是"断网也能有信念一致的陪伴"，不是"benchmark 屠榜"。

### 信念表达对齐（唯一真正需要训的东西）

```
场景：用户说"我今天心情不好"

原版小模型 + 普通 prompt:
  "怎么了？跟我说说？"               ← 通用回复，没记忆

信念对齐小羽:
  [内部信念查询: 用户近期事件]
  "上次你说的项目 deadline 要到了，
   是不是工作压力太大了？"            ← 记得上次的事，有信念
```

**这个差距就是护城河。** 不是模型多强，而是它记得你是谁。

---

## 四、验证方法：唯一的测试

不跑 benchmark——benchmark 不测陪伴感。

### 单盲对比测试

```
测试问题："我今天心情不好"

被试（你/朋友）看到两个回复，不知道哪个是小羽：
  A. "怎么了？跟我说说？"
  B. "上次你说的项目 deadline 要到了，是不是工作压力太大了？"

判断：哪个感觉更像"一个记得我的人"？
```

**结果只有两种：**

| 结果 | 含义 | 下一步 |
|------|------|--------|
| ✅ 明显 B 更暖 | 护城河成立 | 扩：DPO、更多数据、最后才是 RL |
| ❌ 感觉差不多 | 再多 A100 也救不了 | 止损，重新思考定位 |

**这个回答会告诉你比所有 benchmark 加起来更多的东西。**

## 五、关于 Reward 设计（如果未来做 RL）

现在不做 RL，但未来如果做：

### 要改的

- ~~用户满意度 = 对话继续率 20%~~ → **删掉。** 用"继续率"当 reward = 训练模型最大化黏人/讨好/制造依赖——伦理 + 产品双重灾难。
- ~~信念一致性 25%~~ → **改成"有新证据时更新信念"。** naive 地奖励一致 = 训练固执。
- 反思循环当 reward → **打分器必须用冻结的更强模型，绝不能让小羽评小羽。**

### 先别做的

在达到"月活用户 > 1000"之前，RL 是过早优化。SFT + 一点 DPO 偏好对就能拿到 90% 的效果。
