# Pack Runtime 基础蓝图落地路线

> 状态：当前工程化路线图。目标是让云雀继续保持轻内核，把 `doc/` 里的蓝图能力逐步沉淀为可安装、可启停、可回滚、可被前端同步和 SDK 调用的可选 Pack，而不是继续堆进主系统。

## 1. 架构原则

1. **轻内核只保留稳定契约**：核心只负责身份、认证、registry、Pack Runtime、Ledger/状态基础设施和最小 Gateway。业务能力默认进入 pack。
2. **Pack 是新增能力默认形态**：LAA/LoRA、CogniKernel、浏览器意图层、RPA、WASM 插件引擎、异常检测等蓝图能力，如果能独立启停，就先定义 manifest，再接入实现。
3. **前端从后端 registry 同步**：菜单、路由、页面入口、资源 URL、SDK import、backend route method/path 与 manifest capability 索引都来自 `/v1/packs/enabled`、`/v1/packs/backend-modules` 或 `/v1/packs/capabilities`。
4. **SDK slice 优先于大客户端**：外部项目、插件、自动化脚本只导入需要的 `yunque-client/<slice>`；主 `api.ts` 不再新增 pack-domain 方法。
5. **所有大改动可回滚**：manifest 必须声明 `update.rollback=true`，下载包记录 `artifacts`/`previousArtifacts`，工程提交保持小步可回滚。

2026-05-17 更新：Pack Runtime 在 workflow capability plan 之上继续新增只读 `GET /v1/packs/capabilities/prepare?capability=...`，把多能力预检结果转换成企业可执行前的“增量包准备清单”。返回 `use_steps`、`enable_steps`、`install_steps`、`download_steps`、`route_audit_fix_steps`、`package_url`、`manifest_url`、`frontend_url`、`sha256`、`size_bytes` 与原始 `plan`，用于外部 SDK、CI sidecar 和 Pack 控制台明确展示：哪些能力已经可用，哪些只需启用，哪些需要用户选择下载/安装，哪些因为 backend route audit 漂移需要修复。该 prepare 接口仍然 side-effect free，不自动下载、不安装、不启用；真正变更仍走显式 `install` / `enable` / `rollback` 端点，从而保持企业审计和用户选择权。

## 4. 每个蓝图 Pack 的最小交付标准

1. `packs/official/<slug>/pack.json`：包含 `backend.routes`、`backend.routeSpecs`、`frontend.menus`、`frontend.routes`、`sdk.typescript`、`distribution.*`、`update.rollback=true`。

3. `apps/web/src/app/packs/<slug>/page.tsx`：专属页面可以很薄，但必须使用 pack client，不依赖主 `api` 大对象。
4. `apps/web/src/lib/<slug>-pack-client.ts`：前端轻量 client；必要时同步到 `packages/yunque-client/src/<slice>.ts`。
5. 测试与契约：至少覆盖 method/path gate、SDK/client 调用、manifest contract、scaffold contract。
6. 文档：更新 `docs/guide/pack-runtime-state.md` 与中文页，说明该蓝图能力是否已完成最小闭环。
