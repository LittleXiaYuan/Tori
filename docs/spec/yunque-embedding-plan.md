# 云雀嵌入模型方案（务实版）

Status: living plan. 目标：让云雀拥有**自己特化的嵌入模型**支撑「记得你」语义召回，
路线**稳健、现实**——基于成熟开源底座微调，不追极小体积、不对标 SOTA。

## 定位与非目标

- **目标**：稳、可落地、可离线、可逐步特化为「云雀语义」。
- **非目标**：不从零训练（=重造 BGE）；不追 8MB 极小（质量不现实）；不对标/吊打 SOTA。
- **现实体积**：bge-small int8 ≈ 130MB；bge-base ≈ 400MB；bge-m3 ≈ 1–2GB。按部署档位选。

## 底座选型（开源，中文友好）

| 底座 | 参数/维度 | 体积(约) | 适用 |
|---|---|---|---|
| `bge-small-zh-v1.5` | ~33M / 512 | ~130MB(fp32)/~33MB(int8) | 端侧/桌面，资源紧 |
| `bge-base-zh-v1.5` | ~110M / 768 | ~400MB | 均衡（推荐起步） |
| `bge-m3` | ~560M / 1024 | ~1–2GB | 多语/最强召回，服务端 |

起步推荐 **bge-base-zh-v1.5**（质量/体积均衡），端侧档用 bge-small。

## 分阶段（先能用 → 再特化 → 再瘦身）

1. **Phase 1 — 直接用底座（现在，零训练）**
   开源 bge 经 `EMBED_BASE_URL` 接入（代码已支持，含 `/v1` 兼容与 `EMBED_API_KEY`）。
   立刻拿到语义召回；用 `cmd/recall-eval` 测命中率，建立基线。**此阶段无需自训。**
2. **Phase 2 — 云雀特化微调（要 Python+GPU，仓库外）**
   在云雀语料上做对比学习微调底座 → 「云雀特化嵌入」。LoRA 或全量微调。
   产物：微调后权重。
3. **Phase 3 — 服务/瘦身（可选）**
   int8 量化 / ONNX 导出（进程内 `hugot`）/ 或 Model2Vec 静态蒸馏（纯 Go 内置）。
   Matryoshka 可变维度（一份权重支持 256/512/768 维裁剪）作为后续增强。

## 训练数据管线（云雀侧可做）

对比学习需要 (anchor, positive, negative) 三元组：

- **anchor**：用户问句 / 当前消息。
- **positive**：命中的记忆、其改写、同会话被一起召回的条目。
- **negative**：随机记忆 + 难负例（向量近但语义不同）。

来源：`internal/agentcore/memory`（记忆条目）、会话历史、`DataCollector`（反思闭环已在采样）、
`ledger` Recall 的命中/未命中日志。**可做一个导出工具**把这些转成训练集（jsonl）。

## 服务方式（任选，互不影响对话模型）

- **托管/自托管 API**：`EMBED_BASE_URL`（SiliconFlow / 自托管 Ollama·TEI）。
- **进程内 ONNX**：`hugot`/`onnxruntime-go`（引入原生依赖）。
- **纯 Go 静态**：Model2Vec 蒸馏（无原生依赖、可内置单二进制）——最契合云雀，但质量略降，列为后续。

## 评测

- 主指标：**留出集召回命中率**（复用 `cmd/recall-eval`，换说法查 → 命中率）。
- 辅助：小型中文 MTEB 子集（检索/语义相似度）。
- 对照：底座原版 vs 云雀微调版 vs 关键词基线。

## 仓库内 vs 仓库外（诚实边界）

- **仓库内（Go，可立即做）**：嵌入接入/配置（已修）、数据导出管线、Go 加载器/格式（若走 ONNX/静态）、评测工具。
- **仓库外（Python+GPU）**：实际微调/量化/蒸馏/导出权重。本 Go 环境跑不了，提供脚本由 GPU 机执行。

## 落地顺序建议

1. Phase 1 跑通基线（接 bge + recall-eval 出命中率）。
2. 写数据导出工具（造对比对）。
3. 写微调 + 蒸馏 Python 脚本（GPU 机执行）。
4. 微调出云雀特化版 → 再测命中率对照。
5. 视部署档位决定是否量化/静态化 + 服务方式。
